## How to resume (fresh agents)

1. Find the next PR row marked `todo` in the [PRs table](#prs) below; that is your scope.
2. Read [`1231_DESIGN.md`](1231_DESIGN.md), find the section matching the PR title — full spec for the PR is there.
3. Read [`1231_AUDIT.md`](1231_AUDIT.md) for empirical findings: Section 4 = per-attribute classification, Section 5 = validator inventory, Section 4 §Implementation implications = `NewValue` null-Object guidance.
4. Read [`1231_GAP_MATRIX.md`](1231_GAP_MATRIX.md) for the per-attribute target table that gates Phase 2 PRs.
5. For Phase 1 refactor PRs (#3–#5): the "Note" callouts at the top of [`docs/adr/008-sub-block-contract.md`](../docs/adr/008-sub-block-contract.md) (Context section) and [`docs/adr/004-schema-design-conventions.md`](../docs/adr/004-schema-design-conventions.md) §Provider Defaults vs PVE Defaults enumerate the conformance gaps these PRs must close.
6. Skim the [Mid-execution decisions](#mid-execution-decisions) table below for any decisions that affect the PR you're picking up — entries are tagged with `Affects` column.
7. Run `/bpg:start-issue 1231` to scaffold the branch + per-PR session state file (`.dev/1231_<PR>_SESSION_STATE.md`).

## Update protocol

- Bump status when a PR opens, enters review, merges, or blocks.
- Fill `Branch` when branch is created, `PR` when PR is opened, `Session` with the per-PR session-state filename when work begins.
- Recompute the at-a-glance counts on each status change.
- New mid-execution decisions append to the decisions log immediately; design doc absorbs them at phase boundary (end of Phase 1 / end of Phase 2).
- Blockers stay in the active table until resolved; on resolve, fill `Resolved` date and leave the row for history.
